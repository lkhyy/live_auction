## 4. 核心功能清单

1. **主播开播与橱窗管理**：管理端直播控制台创建拍品，配置起拍价、固定加价、封顶价、软关闭延时，支持开播与切品讲解。
2. **买家进房与实时出价**：H5 直播大厅进入直播间，HTTP 出价 + WebSocket 推送排名与毫秒级倒计时。
3. **多品直播场次**：单直播间多件拍品，`activeAuctionId` 标识当前讲解商品，橱窗统一展示。
4. **成交与订单闭环**：竞拍结束或封顶后自动结算生成订单，买家 H5 模拟支付，主播端查看订单。
5. **参与状态跟踪**：「我参与的」按拍品聚合，展示领先/被超越，可跳回直播间继续竞拍。
6. **可配置出价风控**：基于规则的异常出价检测（价格偏离、加价异常等），非大模型推理。

---

## 5. 端到端使用流程

1. 主播在管理端登录，进入直播控制台，为房间添加拍品并配置规则，随后开播或切换讲解中商品。  
2. 买家在用户端 H5 打开直播大厅，选择「直播中」房间进入多品橱窗页。  
3. 买家登录后对当前讲解拍品出价；系统经 Redis Lua 原子校验价格与软关闭延时，经 WebSocket 推送排名与倒计时。  
4. 竞拍到时或达到封顶价后，后端结算并生成订单。  
5. 买家在「我参与的」查看状态，在「订单」中完成模拟支付。  
6. 主播在管理端「订单管理」查看本场成交。  
7. 演示前可执行 `npm run storage:reset` 恢复种子账号与演示房间数据。

## 11. 大模型 / AI 能力使用说明

本系统为直播电商场景的**全栈实时竞拍平台**，核心能力为 Redis Lua 原子出价、Socket.io 实时广播、BullMQ 异步落库。

- **未集成**大语言模型、RAG、向量库或 Agent 工作流。
- **异常出价检测**（`bid-anomaly.util.ts`）为可配置**规则引擎**（价格偏离、加价异常、围标模式等）。
- 直播间背景视频为预录/占位素材（`public/demo/`），文档中提及的 AI 仅指**可选的内容制作工具**，不参与系统运行时推理。

## 12. 关键工程难点与解决方案

### 12.1 高并发出价一致性

**难点**：同场大量买家同时出价，若读写分离或多次 Redis 命令，易出现超价、排名错乱。  
**方案**：出价热路径集中在 **Redis Lua** 单脚本内完成价格校验、ZSet 排名、`seq` 递增与软关闭延时；HTTP 接口不等待 MySQL 写入。  
**效果**：热路径内存计算，配合 [load-tests/k6-bid-load.js](../load-tests/k6-bid-load.js) 可压测 200 次/秒量级（见 [压测报告.md](./压测报告.md)）。

### 12.2 多端时钟与消息乱序

**难点**：客户端时钟不准会导致倒计时漂移；弱网下 WS 消息可能乱序到达。  
**方案**：服务端周期性 `timer_sync` + 客户端 `clockOffset` 校正；广播 payload 带单调 `seq`，客户端丢弃过期 `bid_update`。  
**效果**：H5 与管理端倒计时与服务器对齐，排名展示稳定。

### 12.3 持久化与接口延迟解耦

**难点**：每次出价同步写 MySQL 会拖慢接口，无法满足低延迟。  
**方案**：出价成功后 **BullMQ** 异步入队 `bid-persist`；HTTP 请求带 **Idempotency-Key** 防重复。  
**效果**：接口专注 Redis，落库失败可重试，不影响当前最高价判定。

## 13. 项目亮点 / 创新点

1. **热路径与存储分离**：出价走 Redis Lua，MySQL 异步同步，面向千级同场并发设计。  
2. **直播多品橱窗**：单房间多拍品、讲解中切换，配合「我参与的」状态跟踪，贴近真实直播带货路径。  
3. **软关闭防狙击 + 规则风控**：可配置延时重置与异常出价规则，而非简单倒计时拍卖 demo。
